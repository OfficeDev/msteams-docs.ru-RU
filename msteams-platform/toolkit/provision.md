# <a name="use-teams-toolkit-to-provision-cloud-resources"></a>Использование Teams набор средств для предоставления облачных ресурсов

TeamsFx обеспечивает бесшовную интеграцию с облаком Azure и Microsoft 365, что позволяет разместить приложение в Azure с одной командой. TeamsFx интегрируется с Azure Resource Manager, что позволяет декларативно обеспечить ресурсы Azure, необходимые вашему приложению, используя инфраструктуру в качестве кода.  

## <a name="prerequisites"></a>Предварительные требования

* Предпосылки учетной записи

    Для предоставления облачных ресурсов необходимо иметь следующие учетные записи с надлежащими разрешениями. Дополнительные сведения см. в [записи подготовки учетных записей для создания Teams приложения.](accounts.md)
    * Microsoft 365
    * Azure с допустимой подпиской

* [Установка Teams набор средств](https://marketplace.visualstudio.com/items?itemName=TeamsDevApp.ms-teams-vscode-extension) версии v3.0.0+.

> [!TIP]
> Вы уже должны иметь проект Teams приложения, открытый в коде VS.

## <a name="provision-using-teams-toolkit"></a>Подготовка с Teams набор средств

Подготовка выполняется с одной командой в Teams набор средств для Visual Studio Code или TeamsFx CLI следующим образом:

## <a name="resource-creation"></a>Создание ресурсов

При запуске команды Teams набор средств или TeamsFx CLI можно получить следующие ресурсы:

* AAD приложения под вашим Microsoft 365 клиентом
* Teams приложения в Microsoft 365 платформе Teams клиента
* Ресурсы Azure в выбранной подписке Azure

При создании нового проекта все ресурсы Azure будут созданы. Созданный ARM определяет все ресурсы Azure и помогает создавать необходимые ресурсы Azure во время предоставления. При [добавлении](./add-resource.md) новых возможностей и ресурсов в существующий проект обновленный ARM отображает последние изменения.

## <a name="customize-resource-provision"></a>Настройка предоставления ресурсов

Teams набор средств позволяет использовать инфраструктуру в качестве кода для определения необходимых ресурсов Azure и их настройки. Для определения ресурсов Azure используется ARM шаблон. Шаблон ARM — это набор бицепсных файлов, определяя инфраструктуру и конфигурацию для проекта. Можно настроить ресурсы Azure, созданные путем изменения ARM шаблона. Дополнительные сведения см. в [документе bicep.](/azure/azure-resource-manager/bicep.md) Подготовка с ARM включает изменение следующих двух наборов файлов, параметров и шаблонов:

* ARM файлов параметров () находятся в папке для передачи `azure.parameters.{your_env_name}.json` `.fx/configs` параметров шаблонам.
* ARM шаблонов, расположенных в `templates/azure` этой папке, содержатся следующие файлы:

| File | Что он делает | Разрешить настройку |
| --- | --- | --- |
| main.bicep | Предоставление точки входа для предоставления ресурсов Azure | Да |
| provision.bicep | Создание и присвоение ресурсов Azure | Да |
| config.bicep | Добавление необходимых конфигураций TeamsFx в ресурсы Azure | Да |
| provision/xxx.bicep | создание и присвоение каждого ресурса Azure, потребляемого `provision.bicep` | Да |
| teamsfx/xxx.bicep | Добавление необходимых конфигураций TeamsFx для каждого ресурса Azure, потребляемого `config.bicep`| Нет |

> [!NOTE]
> при добавлении ресурсов или возможностей в проект будет `teamsfx/xxx.bicep` регенерирован. Поэтому он помечен как не настраиваемый. Если у вас действительно есть требование изменить эти бицепсные файлы, рекомендуем использовать Git для отслеживания изменений в файлах, чтобы не потерять изменения при добавлении ресурсов или `teamsfx/xxx.bicep` возможностей.

#### <a name="referencing-environment-variables-in-parameter-files"></a>Переменные среды ссылок в файлах параметров

Иногда может не потребоваться жестко кодировать значения в файлах параметров. Например, если значение является секретом. Файлы параметров поддерживают ссылки на значения переменных среды. Синтаксис в значениях параметров можно использовать для того, чтобы сообщить инструменту, что значение должно быть разрешено из `{{$env.YOUR_ENV_VARIABLE_NAME}}` переменной текущей среды.

В следующем примере будет прочитано значение `mySelfHostedDbConnectionString` параметра из переменной `DB_CONNECTION_STRING` среды:

```json
...
    "mySelfHostedDbConnectionString": "{{$env.DB_CONNECTION_STRING}}"
...
```

#### <a name="customize-arm-template-files"></a>Настройка ARM шаблонов

Если заранее настроенные шаблоны не соответствуют требованиям приложения, можно настроить ARM в `templates/azure` папке. Например, можно настроить шаблон ARM, чтобы создать дополнительные ресурсы Azure для вашего приложения. Это предварительный сценарий и требует, чтобы у вас были базовые знания языка бицепса, который используется для ARM шаблона. Вы можете начать работу с bicep в [документации bicep](/azure/azure-resource-manager/bicep/?branch).
> [!NOTE]
> Шаблон ARM общий для всех сред. Можно использовать [условное развертывание,](/azure/azure-resource-manager/bicep/conditional-resource-deployment?branch) если поведение обеспечения разное между средами.

Чтобы обеспечить правильное обеспечение функций инструментов TeamsFx, убедитесь, что настраиваемый шаблон ARM соответствует следующему требованию. Если вы используете другие средства для дальнейшей разработки, вы можете игнорировать эти требования.

* Сохранить структуру папки и имя файла без изменений. При добавлении дополнительных ресурсов и возможностей в проект этот инструмент может добавлять новое содержимое к существующим файлам.
* Не изменяется имя автоматически созданных параметров, а также имена свойств. Автоматически созданные параметры могут использоваться при добавлении дополнительных ресурсов и возможностей в проект.
* Сохранение вывода автоматического шаблона ARM без изменений. Вы можете добавить дополнительные выходы в ARM шаблон. Выход будет сохраняться и использоваться в других функций, таких как `.fx/states/state.{env}.json` развертывание, проверка файла манифеста и т.д.

### <a name="customization-scenarios"></a>Сценарии настройки

Это некоторые распространенные сценарии, которые можно настроить.

#### <a name="use-an-existing-aad-app-for-your-teams-app"></a>Используйте существующее приложение AAD для Teams приложения

Вы можете добавить следующий фрагмент конфигурации в файл, чтобы использовать AAD приложение, созданное самостоятельно для `.fx/configs/config.{env}.json` Teams приложения. Чтобы создать приложение AAD, <https://aka.ms/teamsfx-existing-aad-doc> см. в .

```json
"auth": {
    "clientId": "<your AAD app client id>",
    "clientSecret": "{{$env.ENV_NAME_THAT_STORES_YOUR_SECRET}}",
    "objectId": "<your AAD app object id>",
    "accessAsUserScopeId": "<id of the access_as_user scope>"
}
```

После добавления выше фрагмента добавьте свой секрет в связанную переменную среды, чтобы инструменты могли устранить фактический секрет во время предоставления.

> [!NOTE]
> Не следует делиться одним AAD приложением в нескольких средах. Если у вас нет разрешения на обновление AAD, вы получите предупреждение с инструкциями о том, как вручную обновить AAD приложение. Следуйте инструкциям по обновлению приложения AAD после предоставления.

#### <a name="use-an-existing-aad-app-for-your-bot"></a>Используйте существующее приложение AAD для бота

Вы можете добавить следующий фрагмент конфигурации в файл, чтобы использовать AAD приложение, созданное `.fx/configs/config.{env}.json` самостоятельно для вашего бота.

```json
"bot": {
    "appId": "<your AAD app client id>",
    "appPassword": "{{$env.ENV_NAME_THAT_STORES_YOUR_SECRET}}"
}
```

После добавления выше фрагмента добавьте свой секрет в связанную переменную среды, чтобы инструменты могли устранить фактический секрет во время предоставления.

> [!NOTE]
> Не следует делиться одним AAD приложением в нескольких средах.

#### <a name="skip-adding-user-for-sql-database"></a>Пропустить добавление пользователя для SQL базы данных

Иногда при попытках добавить пользователя в базу данных SQL может возникнуть недостаточная ошибка разрешения. Вы можете добавить следующий фрагмент конфигурации в файл, чтобы пропустить `.fx/configs/config.{env}.json` добавление SQL пользователя базы данных.

```json
"skipAddingSqlUser": true
```

### <a name="specifying-the-name-of-function-app-instance"></a>Указание имени экземпляра Приложения функции

В этом примере вместо имени по умолчанию укажите другое имя экземпляра `contosoteamsappapi` Приложения функции.

> [!NOTE]
> Если вы уже создали среду, указать имя создаст для вас новый экземпляр Приложения функции, а не переименование экземпляра, созданного ранее.

Следующие действия:

1. Откройте `.fx/configs/azure.parameters.{env}.json` для текущей среды.
2. Добавьте новое свойство `functionAppName` к значению параметра `provisionParameters` .
3. Заполните "contosoteamsappapi" в качестве значения `functionAppName`
4. Конечный файл параметров отображается следующим образом:

    ```json
    {
        "$schema": "https://schema.management.azure.com/schemas/2019-04-01/deploymentParameters.json#",
        "contentVersion": "1.0.0.0",
        "parameters": {
            "provisionParameters": {
            "value": {
                "functionAppName": "contosoteamsappapi"
                ...
                }
            }
        }
    }
    ```

### <a name="scenerio"></a>Scenerio 

**Добавление другого ресурса Azure (хранилище Azure) в приложение**

Рассмотрим сценарий, чтобы добавить служба хранилища Azure в служба хранилища Azure Azure Function для хранения некоторых данных blob. Автопоток для обновления шаблона bicep с помощью служба хранилища Azure поддержки. Однако вы можете изменить бицепс-файл и добавить ресурс. Действия следующие:

1. Создание проекта вкладки
2. Добавьте функцию в проект. Дополнительные информацию о добавлении ресурсов можно посетить в [дополнительных](./add-resource.md) ресурсах.
3. Объявление новой учетной записи служба хранилища в ARM шаблоне. Для упрощения мы объявим ресурс `templates/azure/provision/function.bicep` напрямую. Вы можете свободно объявлять ресурсы в других местах.

    `````````bicep
    var applicationStorageAccountName = 'myapp${uniqueString(resourceGroup().id)}'
    resource applicationStorageAccount 'Microsoft.Storage/storageAccounts@2021-06-01' = {
        name: applicationStorageAccountName
        location: resourceGroup().location
        kind: 'Storage'
        sku: {
            name: 'Standard_LRS'
        }
    }
    `````````

4. Обновим приложение Azure Function App Параметры строкой служба хранилища Azure подключения, которая является тем же файлом `templates/azure/provision/function.bicep` в шаге 3. Добавьте следующий фрагмент в `functionApp` массив `appSettings` ресурса.

    ``````````````````bicep
    {
        name: 'MyAppStorageAccountConnectionString'
        value: 'DefaultEndpointsProtocol=https;AccountName=${applicationStorageAccount.name};AccountKey=${listKeys(applicationStorageAccount.id, '2021-06-01').keys[0].value}'
    }
    ```````````````````

5. Теперь вы можете обновить функцию с помощью привязки служба хранилища Azure вывода.
