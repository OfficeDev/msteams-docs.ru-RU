# <a name="use-teams-toolkit-to-provision-cloud-resources"></a>Использование Teams набор средств для предоставления облачных ресурсов

TeamsFx интегрируется с облаком Azure и Microsoft 365, что позволяет разместить приложение в Azure с одной командой. TeamsFx интегрируется с Azure Resource Manager, что позволяет обеспечить ресурсы Azure, которые нужны вашему приложению для подхода к коду.  

## <a name="prerequisites"></a>Предварительные условия

* Необходимые условия учетной записи для предоставления облачных ресурсов должны иметь следующие учетные записи:

  * Microsoft 365 учетной записи с допустимой подпиской
  * Azure с допустимой подпиской Для получения дополнительных сведений см. сведения о подготовке учетных записей для [создания Teams приложения](accounts.md).

* [Установка Teams набор средств](https://marketplace.visualstudio.com/items?itemName=TeamsDevApp.ms-teams-vscode-extension) версии v3.0.0+.

> [!TIP]
> Убедитесь, что Teams проекта приложения открыт в коде VS.

## <a name="provision-using-teams-toolkit"></a>Подготовка с Teams набор средств

Подготовка выполняется с одной командой в Teams набор средств для Visual Studio Code или TeamsFx CLI следующим образом:

[Подготовка приложения на основе Azure](/microsoftteams/platform/sbs-gs-javascript?tabs=vscode%2Cvsc%2Cviscode%2Cvcode&tutorial-step=8&branch)

## <a name="resource-creation"></a>Создание ресурсов

При запуске команды Teams набор средств или TeamsFx CLI можно получить следующие ресурсы:

* Microsoft Azure Active Directory (Azure AD) под вашим Microsoft 365 клиентом
* Teams приложения в Microsoft 365 платформе Teams клиента
* Ресурсы Azure в выбранной подписке Azure

При создании нового проекта можно использовать все ресурсы Azure. Шаблон ARM определяет все ресурсы Azure и помогает создавать необходимые ресурсы Azure во время предоставления. При [добавлении нового](./add-resource.md) ресурса возможностей в существующий проект обновленный шаблон ARM отражает последние изменения.

> Дополнительные сведения о оценке затрат на услуги Azure см. в [калькуляторе ценообразования](https://azure.microsoft.com/pricing/calculator/).

## <a name="customize-resource-provision"></a>Настройка предоставления ресурсов

Teams набор средств позволяет использовать инфраструктуру в качестве кода для определения необходимых ресурсов Azure и настройки. Средство использует ARM для определения ресурсов Azure. Шаблон ARM — это набор бицепсных файлов, определяя инфраструктуру и конфигурацию для проекта. Можно настроить ресурсы Azure, изменяя ARM шаблона. Дополнительные сведения см. в [документе bicep](/azure/azure-resource-manager/bicep.md).

Подготовка с ARM включает изменение следующих наборов файлов, параметров и шаблонов:

* ARM файлов параметров (`azure.parameters.{your_env_name}.json`), расположенных в `.fx/configs` папке, для передачи параметров шаблонам.
* ARM шаблонов, расположенных в `templates/azure`этой папке, содержатся следующие файлы:

| Файл | Функция | Разрешить настройку |
| --- | --- | --- |
| main.bicep | Предоставление точки входа для предоставления ресурсов Azure | Да |
| provision.bicep | Создание и настройка ресурсов Azure | Да |
| config.bicep | Добавление необходимых конфигураций TeamsFx в ресурсы Azure | Да |
| provision/xxx.bicep | Создание и настройка каждого ресурса Azure, потребляемого `provision.bicep` | Да |
| teamsfx/xxx.bicep | Добавление необходимых конфигураций TeamsFx для каждого ресурса Azure, потребляемого `config.bicep`| Нет |

> [!NOTE]
> При добавлении ресурсов `teamsfx/xxx.bicep` или возможностей в проект будет регенерирован, вы не можете настроить то же самое. Чтобы изменить бицепсные файлы, можно использовать Git `teamsfx/xxx.bicep` для отслеживания изменений в файлах, что позволяет не терять изменения при добавлении ресурсов или возможностей.

#### <a name="referencing-environment-variables-in-parameter-files"></a>Переменные среды ссылок в файлах параметров

Если вы не хотите жестко кодировать значения в файлах параметров, например, если значение является секретным. Файлы параметров поддерживают ссылки на значения переменных среды. Синтаксис `{{$env.YOUR_ENV_VARIABLE_NAME}}` в значениях параметров для средства можно использовать для решения из текущей переменной среды.

В следующем примере читается значение `mySelfHostedDbConnectionString` параметра из переменной среды `DB_CONNECTION_STRING`:

```json
...
    "mySelfHostedDbConnectionString": "{{$env.DB_CONNECTION_STRING}}"
...
```

#### <a name="customize-arm-template-files"></a>Настройка ARM шаблонов

Если заранее настроенные шаблоны не соответствуют требованиям приложения, можно настроить ARM в папке `templates/azure` . Например, можно настроить шаблон ARM, чтобы создать дополнительные ресурсы Azure для вашего приложения. Необходимо иметь базовые знания языка бицепса, который используется для ARM шаблона. Вы можете начать работу с bicep в [документации bicep](/azure/azure-resource-manager/bicep/?branch).

> [!NOTE]
> Шаблон ARM общий для всех сред. Можно использовать [условное развертывание,](/azure/azure-resource-manager/bicep/conditional-resource-deployment?branch) если поведение обеспечения различается между средами.

Чтобы обеспечить правильное работу инструмента TeamsFx, настройте ARM шаблон, который соответствует следующему требованию. Если вы используете другой инструмент для дальнейшей разработки, вы можете игнорировать эти требования.

* Сохранить структуру папки и имя файла без изменений. При добавлении дополнительных ресурсов или возможностей в проект этот инструмент может добавлять новый контент к существующим файлам.
* Не изменяется имя автоматически созданных параметров, а также имена свойств. Автоматически созданные параметры могут использоваться при добавлении дополнительных ресурсов или возможностей в проект.
* Сохранение вывода автоматического шаблона ARM без изменений. Дополнительные выходные данные можно добавить в ARM шаблона. Вывод используется и `.fx/states/state.{env}.json` может использоваться в других функций, таких как развертывание, проверка файла манифеста.

### <a name="customization-scenarios"></a>Сценарии настройки

Можно настроить следующие сценарии:

#### <a name="use-an-existing-azure-ad-app-for-your-bot"></a>Использование существующего приложения Azure AD для бота

Вы можете добавить следующий `.fx/configs/config.{env}.json` фрагмент конфигурации в файл, чтобы использовать приложение Azure AD, созданное самостоятельно для Teams приложения. Чтобы создать приложение Azure AD, см. в .<https://aka.ms/teamsfx-existing-aad-doc>

```json
"auth": {
    "clientId": "<your Azure AD app client id>",
    "clientSecret": "{{$env.ENV_NAME_THAT_STORES_YOUR_SECRET}}",
    "objectId": "<your Azure AD app object id>",
    "accessAsUserScopeId": "<id of the access_as_user scope>"
}
```

После добавления фрагмента добавьте свой секрет в связанную переменную среды, чтобы инструмент может разрешить фактический секрет во время предоставления.

> [!NOTE]
> Убедитесь, что одно и то же приложение Azure AD не будет делиться в нескольких средах. Если у вас нет разрешения на обновление приложения Azure AD, вы можете получить предупреждение с инструкциями об обновлении приложения Azure AD вручную. Следуйте инструкциям по обновлению приложения Azure AD после предоставления.

#### <a name="use-an-existing-azure-ad-app-for-your-teams-app"></a>Используйте существующее приложение Azure AD для Teams приложения

Вы можете добавить следующий фрагмент конфигурации `.fx/configs/config.{env}.json` в файл, чтобы использовать приложение Azure AD, созданное самостоятельно для вашего бота:

```json
"bot": {
    "appId": "<your Azure AD app client id>",
    "appPassword": "{{$env.ENV_NAME_THAT_STORES_YOUR_SECRET}}"
}
```

После добавления предыдущего фрагмента добавьте свой секрет в связанную переменную среды для средства для устранения фактического секрета во время предоставления.

#### <a name="skip-adding-user-for-sql-database"></a>Пропустить добавление пользователя для SQL базы данных

Если у вас недостаточно ошибок разрешения при попытках добавить пользователя в базу данных SQL, `.fx/configs/config.{env}.json` можно добавить в файл следующий фрагмент конфигурации, чтобы пропустить добавление SQL пользователя базы данных:

```json
"skipAddingSqlUser": true
```

### <a name="specifying-the-name-of-function-app-instance"></a>Указание имени экземпляра Приложения функции

Вы можете использовать для `contosoteamsappapi` экземпляра приложения функции вместо имени по умолчанию.

> [!NOTE]
> Если вы уже создали среду, указав имя, можно создать для вас новый экземпляр приложения-функции, а не переименовать экземпляр, созданный ранее.

Следующие действия:

1. Откройте `.fx/configs/azure.parameters.{env}.json` для текущей среды.
2. Добавьте новое свойство `functionAppName` к значению параметра `provisionParameters`.
3. Ввод `contosoteamsappapi` в качестве значения `functionAppName`
4. Файл окончательного параметра показан в следующем фрагменте:

    ```json
    {
        "$schema": "https://schema.management.azure.com/schemas/2019-04-01/deploymentParameters.json#",
        "contentVersion": "1.0.0.0",
        "parameters": {
            "provisionParameters": {
            "value": {
                "functionAppName": "contosoteamsappapi"
                ...
                }
            }
        }
    }
    ```

### <a name="scenerio"></a>Scenerio

Добавление другого ресурса Azure или хранилища в приложение:

Рассмотрим сценарий, чтобы сохранить данные blob, необходимо добавить хранилище Azure в свою спинку функции Azure. Автопоток для обновления шаблона бицепса с помощью службы хранения Azure не существует. Однако вы можете изменить бицепс-файл и добавить ресурс. Для этого выполните следующие действия.

1. Создание проекта вкладки.
2. Добавление функции в проект. Дополнительные сведения см. в [добавлении ресурсов](./add-resource.md).
3. Объявите новую учетную запись хранилища в ARM шаблоне. Вы можете объявить ресурс напрямую `templates/azure/provision/function.bicep` . Вы можете свободно объявлять ресурсы в других местах.

    `````````bicep
    var applicationStorageAccountName = 'myapp${uniqueString(resourceGroup().id)}'
    resource applicationStorageAccount 'Microsoft.Storage/storageAccounts@2021-06-01' = {
        name: applicationStorageAccountName
        location: resourceGroup().location
        kind: 'Storage'
        sku: {
            name: 'Standard_LRS'
        }
    }
    `````````

4. Обновление параметров приложения-функции Azure с помощью строки подключения к хранилищам Azure `templates/azure/provision/function.bicep`. Добавьте следующий фрагмент в `functionApp` массив ресурса `appSettings` :

    ``````````````````bicep
    {
        name: 'MyAppStorageAccountConnectionString'
        value: 'DefaultEndpointsProtocol=https;AccountName=${applicationStorageAccount.name};AccountKey=${listKeys(applicationStorageAccount.id, '2021-06-01').keys[0].value}'
    }
    ```````````````````

5. Вы можете обновить функцию с помощью привязки к выводам хранилища Azure.

<details>

<summary><b>Как устранить неполадки?</b></summary>

Если вы получаете ошибки с Teams набор средств в Visual Studio Code, вы можете выбрать Техническая поддержка в уведомлении об ошибке, чтобы перейти к связанному документу. Если вы используете CLI TeamsFx, в конце сообщения об ошибке будет гиперссылка, которая указывает на справку. Вы также можете просматривать [справку о предоставлении непосредственно](https://aka.ms/teamsfx-arm-help) .

<br>

</details>

<details>

<summary><b>Как перейти на другую подписку Azure во время подготовка?</b></summary>

1. Переключение подписки в текущей учетной записи или выход и выбор новой подписки.
2. Если вы уже создали текущую среду, необходимо создать новую среду и выполнить предоставление, так как ARM не поддерживает перемещение ресурсов.
3. Если вы не выработали текущую среду, вы можете запускать предоставление непосредственно.

<br>

</details>

<details>

<summary><b>Как изменить группу ресурсов при подготовках?</b></summary>

Перед предоставлением средство спросит вас, хотите ли вы создать новую группу ресурсов или использовать существующую. На этом шаге можно уступить новое имя группы ресурсов или выбрать существующее.

<br>

</details>
