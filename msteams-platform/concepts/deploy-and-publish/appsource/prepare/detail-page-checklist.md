# <a name="create-a-store-listing-for-your-microsoft-teams-app"></a>Создание списка магазина для Microsoft Teams приложения

Сведения, которые вы [](https://partner.microsoft.com) передаете в Центр партнеров&#8212;включая ваше имя, описания, значки и изображения,&#8212;становятся Microsoft Teams магазина и списка Microsoft AppSource для вашего приложения.

Список магазинов может быть первым впечатлением от вашего приложения. Увеличение установок с помощью списка, который эффективно передает преимущества, функциональные возможности и бренд вашего приложения.

## <a name="specify-a-short-name"></a>Укажите короткое имя

Имя вашего приложения (в частности, его короткое [*имя)*](~/resources/schema/manifest-schema.md#name)играет решающую роль в том, как пользователи обнаруживают его в магазине.

В следующем примере выделяется, где короткое имя приложения отображается в списке магазина.

:::image type="content" source="../../../../assets/images/store-detail-page/AppName-02.png" alt-text="Пример экрана показывает, где короткое имя приложения отображается в списке магазина.":::

### <a name="best-practices-for-names"></a>Лучшие практики для имен

**Рекомендуется:**

* Выберите простое запоминающееся имя, которое подсказает, что делает ваше приложение.
* Будьте отличительными.
* Избегайте опечаток и грамматических ошибок.

**Не рекомендуется:**

* Используйте ненормативную или пренебожную лексику.
* Используйте расово или культурно нечувствительный язык.
* Используйте общие термины или имена, аналогичные существующим приложениям.
* Включай Teams, "Microsoft", существующие и предстоящие имена продуктов Майкрософт или "приложение" в имя.

> [!NOTE]
> Если ваше приложение является частью официального партнерства с Microsoft, имя приложения должно быть первым (например, *Connector Salesforce для Microsoft Teams).*

## <a name="write-descriptions"></a>Написание описаний

Вам нужно краткое и длинное описание приложения.

### <a name="short-description"></a>Краткое описание

Краткий сводка приложения, которая должна быть оригинальной, привлекательной и направленной для целевой аудитории. В идеале сохраняем краткое описание в одном предложении.

В следующем примере выделяется, где краткое описание приложения отображается в списке магазина:

:::image type="content" source="~/assets/images/store-detail-page/ShortDescription-02.png" alt-text="Пример экрана показывает, где краткое описание приложения отображается в списке магазина.":::

#### <a name="best-practices-for-short-descriptions"></a>Лучшие практики для коротких описаний

**Рекомендуется:**

* Сначала разместите наиболее важные сведения.
* Включай ключевые слова, которые клиенты, скорее всего, будут искать.

**Не рекомендуется:**

* Повторите имя приложения.
* Использование жаргона или специализированной терминологии. (Вы не можете предположить, что пользователи знают, что искать.)

### <a name="long-description"></a>Подробное описание

Длинное описание может предоставить привлекательный повествование, которое выделяет основные функции приложения, проблемы, которые оно решает, и его целевую аудиторию. Хотя это описание может быть до 4000 символов, большинство пользователей будут читать только между 300-500 слов.

В следующем примере выделяется, где длинное описание приложения отображается в списке магазина:

:::image type="content" source="~/assets/images/store-detail-page/ShortDescription-02.png" alt-text="Пример экрана показывает, где длинное описание приложения отображается в списке магазина.":::

#### <a name="usage-examples"></a>Примеры использования

Следующие фразы являются примерами того, что разрешено при написании длинных описаний:

* "<имя приложения *>* с Microsoft Teams"
* "... тип <*для>* для Microsoft Teams"
* "<*имя* приложения> интегрируется с Microsoft Teams"
* "... интегрирована с Microsoft Teams"
* "... для пользователей, работающих с Microsoft Teams"
* "... для <*задачи>* в Microsoft Teams"
* "... построено на ..."
* "... работает на ..."
* "... включено ..."
* "... разработано для ..."
* "... предназначен для ..."

#### <a name="best-practices-for-long-descriptions"></a>Лучшие практики для длинных описаний

**Рекомендуется:**

* Используйте [Markdown для](https://support.office.com/article/use-markdown-formatting-in-teams-4d10bd65-55e2-4b2d-a1f3-2bebdcd2c772) формата описания.
* Список функций с точками пули, поэтому проще сканировать описание.
* Используйте активный голос и поговорите с пользователями напрямую (например, *Вы можете ...*).
* Включаем ссылку справка или поддержка.
* Определите, если это применимо: ограничения, настройка сведений, зависимостей учетных записей и обновления выпуска.

**Не рекомендуется:**

* Более 500 слов.
* Включай слишком много ключевых слов. (Это отвлекает и не поможет людям найти ваше приложение.)
* Используйте следующий язык, если приложение не прошло официальный процесс сертификации:
  * "... сертифицировано для ..."
  * " ... питание от ..."

### <a name="best-practices-for-all-descriptions"></a>Лучшие практики для всех описаний

**Рекомендуется:**

* Ссылки на имена продуктов Майкрософт только при необходимости. Дополнительные сведения о руководстве см. в [руководстве по товарным знакам и брендам Майкрософт.](https://www.microsoft.com/en-us/legal/intellectualproperty/trademarks/usage/general)
* Если вам нужно ссылаться **на Teams,** напишите первую ссылку как **Microsoft Teams**. Последующие ссылки можно сократить до **Teams.**
* Обратитесь **Microsoft 365** вместо **Office 365**.
* Избегайте опечаток и грамматических ошибок.
* Избегайте ненужных капитализаций (например, **пользователи,** а не **пользователи).**
* Избегайте аббревиатур.

**Не рекомендуется:**

* Сокращение Microsoft в качестве **MS или** **MSFT**.
* Указать, что приложение — это предложение от Корпорации Майкрософт, в том числе с помощью слоганы или taglines Майкрософт.
* Используйте имена брендов, защищенные авторским правом, которые не принадлежат вам.

## <a name="adhere-to-icon-design-guidelines"></a>Соблюдайте рекомендации по проектированию значков

Значки являются одним из основных элементов, которые пользователи видят при просмотре магазина. Значки должны сообщать о цели бренда вашего приложения, а также о Teams требованиях.

Дополнительные сведения см. в [специальном руководстве по разработке значков Teams приложений.](~/concepts/build-and-test/apps-package.md#app-icons)

## <a name="capture-screenshots"></a>Захват скриншотов

Скриншоты предоставляют видную визуальную предварительную версию приложения в дополнение к имени, значку и описаниям приложения.

### <a name="requirements-for-screenshots"></a>Требования к скриншотам

* До пяти скриншотов в списке.
* Поддерживаемые типы файлов включают PNG, JPEG и GIF.
* Размеры должны быть 1366x768 пикселей.
* Максимальный размер 1024 КБ.

### <a name="best-practices-for-screenshots"></a>Лучшие практики для скриншотов

**Рекомендуется:**

* Сосредоточься на возможностях приложения (например, на том, как люди могут общаться с вашим ботом).
* Включите содержимое, которое точно представляет ваше приложение.
* Используйте текст разумно.
* Скриншоты кадра с цветом, который отражает ваш бренд и включает маркетинговый контент, аналогичный следующему примеру [Freshdesk](https://appsource.microsoft.com/product/office/WA104381505?src=office&tab=Overview) (требования к размеру применяются к всему изображению, а не только к скриншоту): пример экрана сторонних приложений   :::image type="content" source="../../../../assets/images/freshdesk.png" alt-text="Freshdesk":::

**Не рекомендуется:**

* Показ определенных устройств, например телефонов или ноутбуков.
* Отображение хрома или пользовательского интерфейса, не в приложении.
* Захват любого пользовательского Teams браузера на скриншотах.
* Включите макеты, неточно отражающие фактический пользовательский интерфейс приложения, например отображение приложения в браузере вместо вкладки Teams.

Дополнительные практические действия см. в [иллюстрации craft effective images for Microsoft app stores.](/office/dev/store/craft-effective-appsource-store-images)

## <a name="create-a-video"></a>Создание видео

Видео может быть наиболее эффективным способом общения, почему люди должны использовать ваше приложение. В видео необходимо ответить на следующие вопросы:

* Кто для вашего приложения?
* Какие проблемы может решить приложение?
* Как работает ваше приложение?
* Какие еще преимущества вы получаете от использования приложения?

Если вы включаете видео, оно отображается перед снимками экрана в списке.

### <a name="best-practices-for-videos"></a>Лучшие практики для видео

* Сохраняйте видео в течение 30-90 секунд.
* Стремиться к качеству. В списке пользователи увидят ваше видео перед снимками экрана.
