# <a name="web-apps"></a>Веб-приложения 

Вы можете сделать веб-приложения подходящими с социальными и совместными функциями Teams, правильно интегрируя их с Teams.
  
Различные типы приложений, которые можно интегрировать с Teams, являются следующими:
* **Автономные приложения.** Отдельное приложение — это одно-страница или большое и сложное приложение. Пользователь может использовать некоторые его аспекты в Teams.
* **Приложения для совместной** работы: приложение, уже построенное для социальных и совместных функций, присущих Teams.
* **SharePoint:** страница SharePoint, которая должна всплыть в Teams.

Вы можете составить карту и следовать соответствующим рекомендациям, применимым к сценарию интеграции.
В этом документе представлен обзор возможностей Teams, требований к точкам обмена данными для хранения файлов и данных, требований API, проверки подлинности и глубокой связи приложения с Teams.

## <a name="understand-sharepoint-requirements-and-options"></a>Понимание требований и параметров SharePoint

***Сценарии интеграции:** SharePoint*

Чтобы интегрировать существующую [страницу SharePoint](https://docs.microsoft.com/MicrosoftTeams/teams-standalone-static-tabs-using-spo-sites) в вкладку Teams, необходимо рассмотреть следующее:

* Это должна быть *современная веб-страница* SharePoint.
* Поддерживаются только личные вкладки. Вы не можете интегрировать страницу в качестве вкладки канала.

Кроме того, вы можете создать вкладку Teams [с помощью SharePoint Framework.](https://docs.microsoft.com/sharepoint/dev/spfx/integrate-with-teams-introduction)

## <a name="understand-authentication-options"></a>Понимание параметров проверки подлинности

***Сценарии интеграции:** автономные приложения, приложения для совместной работы, SharePoint*

Azure Active Directory (AD) является поставщиком удостоверений для Teams. Если в вашем приложении используется другой поставщик удостоверений, необходимо либо сделать упражнение сопоставления удостоверений, либо объединить его с Azure AD.

Teams имеет механизмы единого входного (SSO) с Azure AD для сторонних приложений. Он также предоставляет рекомендации по проверке подлинности для других поставщиков удостоверений, использующих такие стандарты, как OAuth и Open ID Connect, известные как OIDC.

Для страниц SharePoint можно использовать только SSO и не может добавить другой Azure AD ID, если вы хотите, чтобы SSO работало для другого приложения, так как ID — это приложение SharePoint.

Дополнительные информацию о [проверке подлинности в Teams](../concepts/authentication/authentication.md).

## <a name="be-smart-when-messaging-users"></a>Будьте умны, когда пользователи сообщений

***Сценарии интеграции:** автономные приложения, приложения для совместной работы, SharePoint*

Используйте бот [в](../bots/what-are-bots.md) приложении Teams для многопотокого разговора, так как он обеспечивает больше гибкости, чем [веб-ок.](../webhooks-and-connectors/what-are-webhooks-and-connectors.md)

Боты также позволяют отправлять активные **сообщения** отдельным пользователям или каналам. Проактивные сообщения — это незапланированные сообщения, запускаемые внешним событием, а не сообщения, отправленные боту. Например, бот отправляет приветствие при установке или при подступах к каналу нового пользователя. 

Отправка упреждающих сообщений требует идентификаторов, определенных для Teams. Вы можете захватить эти сведения, [извлекая](../bots/how-to/get-teams-context.md#fetch-the-roster-or-user-profile)данные реестра или профиля [пользователей,](../bots/how-to/conversations/subscribe-to-conversation-events.md)записавшись на события беседы или используя [Microsoft Graph.](https://docs.microsoft.com/graph/teams-proactive-messaging)

Не спамить пользователей с избыточными сообщениями. Если возможности Teams поддерживают его, пользователи могут настроить параметры уведомлений для вашего приложения.

## <a name="use-sharepoint-for-file-and-data-storage"></a>Использование SharePoint для хранения файлов и данных

***Сценарии интеграции:** Автономные приложения, приложения для совместной работы, страницы SharePoint*
